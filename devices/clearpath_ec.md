# Teknic ClearPath-EC (models ECSK / ECHP)

**Fingerprint**: VID 0x0C96 (3222, Teknic) / PID 0x0001 / Revision 2 / CiA-402 profile 0x192, profile version 0x00040100. DC `AssignActivate` word is **0x0300** for the DC-Sync OpMode and 0x0000 for the Free Run / SM-Synchron OpMode (confirmed from `ethercat/esi/Teknic_ClearPathEC_ESI.xml` lines 8174-8189). CoE mailbox advertises `SdoInfo`, `PdoAssign`, `PdoConfig`, `CompleteAccess`, `SegmentedSdo`, `DiagHistory`, and FoE, so the master can rely on complete-access reads, segmented transfers for strings, and 0x10F3 diagnosis-history upload. SyncManager defaults from ESI: SM0 MBoxOut 0x1000/246B (ctrl 0x26), SM1 MBoxIn 0x1400/246B (ctrl 0x22), SM2 Outputs 0x1800/10B (ctrl 0x64), SM3 Inputs 0x1C00/16B (ctrl 0x20). Minimum cycle time 250 us, must be an integer multiple of 250 us; interpolation time period (0x60C2) MUST equal master cycle. DC-Sync is strongly recommended; SM-Sync accepted; Free Run is explicitly unsupported. The ESI leaves `CycleTimeSync0 / ShiftTimeSync0 / CycleTimeSync1 / ShiftTimeSync1 = 0`, so the master owns those values at runtime. Slave-published DC shift values (from 0x1C32/0x1C33): output delay 440 ns, input delay 5200 ns - anything sub-1 ms must tune sync0 shift so the pulse lands *after* the datagram finishes copying.

**ESI layout facts the master must honour**: the ESI declares three FMMUs (Outputs, Inputs, MBoxState), so the master has to program all three — wiring up only two will pass AL=INIT but fail at SafeOp. Mailbox timeouts are `RequestTimeout=2000` and `ResponseTimeout=20000` (ms); both are generous but a real master should still arm its own upper bound. Every PDO carries `Fixed="0"`, i.e. the drive accepts entry-level remapping in PREOP — we are not limited to reassigning pre-baked mappings. The TwinCAT `AlternativeSmMapping` table inside the ESI gives the canonical mode→PDO pair, with **CSP (0x1601/0x1A01) flagged `Default="1"`** — so the drive's *boot-default* PDO pair is not the combined 0x1600/0x1A00, it is the CSP-only pair. Master code that assumes the combined set without an explicit `0x1C12/0x1C13` rewrite is operating on stale assumptions. Finally: the ESI also carries a second deprecated device record, `ClearPath EC (CoE Rev 1)` with `RevisionNo=#x1`. Our TSV targets Rev 2 (0x1018:3 = 2); the master should refuse Rev 1 so we don't silently mis-apply CoE Rev 2 PDO maps to an older drive.

**Mailbox / CoE capabilities** (ESI line 8171): `SdoInfo`, `PdoAssign`, `PdoConfig`, `CompleteAccess`, `SegmentedSdo`, `DiagHistory`, plus FoE. CompleteAccess lets the master read a whole multi-sub-index object (e.g. all of 0x1018) in one transfer. SegmentedSdo is mandatory for the string objects 0x1008 / 0x1009 / 0x100A (21/3/10 bytes respectively — all exceed the 4-byte expedited window). DiagHistory means 0x10F3 is queryable for stored alarms. FoE is available for firmware upload if we ever need it.

**Diagnosis history strings come from the ESI, not from us.** The `DiagMessages` block in `Teknic_ClearPathEC_ESI.xml` carries 53 English entries with TextIds in the 0xABxx / 0xACxx range (examples: 0xAB02 "FW Problem: type 0", 0xAB1C "E-Stop", 0xAC00 "Errors have been cleared"). When we read 0x10F3:n and it reports a TextId, resolve it against this table — don't hand-translate. This is separate from the 12 short-form error codes in PDF Appendix B (0x21D2 etc.), which come back through 0x603F; both channels coexist.

## Motion features that should shape the miniOS motion kernel

**Home-on-hard-stop is a first-class method (0x6098 = -1, -2).** ClearPath measures motor torque against `Hardstop Torque Maximum` (0x216B, 0.1 percent units) while creeping at Slow Homing Speed (0x6099:2), then backs off by `Physical Home Clearance` (0x2201). Our homing state machine cannot just be limit-switch plus index pulse - it needs a torque-threshold branch with a settable peak, a creep speed, and a back-off distance. Eleven homing methods are exposed (0x60E3) including shaft-angle homing (33, 34) and current-position capture (37, the default). Plan on a homing engine that composes "approach direction / edge type / trigger source / backoff" rather than hard-coding CiA standard numbers.

**Servo-on is a two-step handshake, not a single bit.** Controlword 0x6040 requires bits 1 and 2 to be raised first (Ready-to-Switch-On), *then* bits 0-3 together (Operation Enabled) - raising all four at once is not documented as legal. Add to that: Halt (bit 8) is valid only in PP/PV/TQ/HM and must not be used in CSP/CSV/CST; external limit bits 11/12 are ignored in CST/TQ. Our motion API needs per-mode controlword masks, a small FSA driver that walks Switch-On-Disabled -> Ready -> Switched-On -> Operation-Enabled, and a mode-change sequencer because homing is entered by switching 0x6060 to 6 and back to 8 afterward.

**Real-time constraints and safety hooks are tighter than generic CiA-402.** In CSP the drive *always* abrupt-stops on connection loss regardless of `Abort Connection Option` (0x6007). Five independent stop actions are configurable (Quick Stop 0x605A, Disable 0x605C, Halt 0x605D, Fault Reaction 0x605E, Abort 0x6007) with Position Hold, Dynamic Brake, Coast, and Decel choices; `Delay Disable Time` (0x2170) gates the actual servo-off so an external brake can engage - the motion kernel must model brake latency. Teknic-specific niceties worth surfacing up front: Move Done Torque Foldback (0x2163 + app-config bit 23) for cool running at rest, RMS Level (0x230F) thermal load you can read back, Touch Probe x2 with zero-pulse triggering (0x60B8 default 0x1713) for on-the-fly capture, Overspeed Timeout (0x231A, 10 ms default) that only trips in CSP/CST/TQ, and a full tuning object range 0x2143-0x215D (Kp/Kv/Ki/Kfv/Kfa/Kfj/Knv plus RAS anti-hunt delay 0x2039) that we should expose as a single "drive tuning block" rather than per-parameter. Software position limits (0x607D) are disabled until Has-Homed is true - the kernel must treat pre-home moves differently.
